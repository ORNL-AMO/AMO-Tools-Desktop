# Process Cooling Assessment — Data Model

All types live in `src/app/shared/models/process-cooling-assessment.ts` unless otherwise noted.

---

## Top-Level Shape

```
Assessment (IndexedDB record)
└── processCooling: ProcessCoolingAssessment
    ├── name: string
    ├── setupDone: boolean
    ├── selected: boolean
    ├── isValid: boolean
    ├── existingDataUnits: 'Imperial' | 'Metric'
    ├── systemBasics: SystemBasics
    ├── systemInformation: SystemInformation
    ├── inventory: ChillerInventoryItem[]
    ├── weeklyOperatingSchedule: WeeklyOperatingSchedule
    ├── monthlyOperatingSchedule: MonthlyOperatingSchedule
    ├── modifications: Modification[]
    ├── selectedModificationId: string
    └── weatherData: WeatherContextData (optional)
```

## SystemBasics

Minimal assessment-level metadata captured in the Assessment Settings step.

```typescript
{
  utilityType: string;   // always 'Electricity' for this assessment type
  notes?: string;
}
```

## SystemInformation

Captures the physical plant configuration. Nested under `processCooling.systemInformation`.

```typescript
{
  co2SavingsData: Co2SavingsData;        // emissions tracking (zipcode, eGrid region, etc.)
  operations: OperationsInput;
  airCooledSystemInput: AirCooledSystemInput;    // only used when condenserCoolingMethod = 1 (Air)
  waterCooledSystemInput: WaterCooledSystemInput; // only used when condenserCoolingMethod = 0 (Water)
  towerInput: TowerInput;
  chilledWaterPumpInput: PumpInput;
  condenserWaterPumpInput: PumpInput;
}
```

### OperationsInput

```typescript
{
  annualOperatingHours: number;
  fuelCost: number;
  electricityCost: number;
  chilledWaterSupplyTemp: number;  // °F in Imperial, °C in Metric
  condenserCoolingMethod: CondenserCoolingMethod;  // 0 = Water, 1 = Air
  doChillerLoadSchedulesVary?: boolean;
}
```

### CondenserCoolingMethod

The `condenserCoolingMethod` on `OperationsInput` drives which condenser sub-form is shown:

| Value | Form shown | Type |
|---|---|---|
| `0` | `WaterCooledComponent` | `WaterCooledSystemInput` |
| `1` | `AirCooledComponent` | `AirCooledSystemInput` |

### TowerInput

```typescript
{
  usesFreeCooling: boolean;
  isHEXRequired: boolean;
  HEXApproachTemp: number;
  numberOfTowers: number;
  numberOfFans: number;
  fanSpeedType: number;
  towerSizeMetric: TowerSizeMetric;  // Tons | HP | Unknown
  fanType: FanType;                   // Axial | Centrifugal
  towerSize: number;
  towerType: TowerType;               // e.g. OneCellOneSpeed, VariableSpeed, etc.
}
```

### PumpInput

Used for both chilled-water and condenser-water pumps.

```typescript
{
  variableFlow: boolean;
  flowRate: number;      // gpm (Imperial) or m³/s (Metric)
  efficiency: number;    // %
  motorSize: number;     // HP (Imperial) or kW (Metric)
  motorEfficiency: number; // %
}
```

## ChillerInventoryItem

One per physical chiller in the plant. The assessment requires at least one.

```typescript
{
  itemId: string;                     // UUID, generated by getNewIdString()
  name: string;
  description?: string;
  modifiedDate: Date;
  chillerType: CompressorChillerTypeEnum;  // CENTRIFUGAL | RECIPROCATING | SCREW
  capacity: number;                   // tons (Imperial) or kW (Metric)
  isFullLoadEfficiencyKnown: boolean;
  fullLoadEfficiency: number;         // kW/ton (Imperial) or kW/kW (Metric)
  age: number;                        // years
  installVSD: boolean;
  refrigerantType: RefrigerantType;   // R11 | R123 | R12 | R134a | R22 | R717
  isCustomChiller: boolean;
  useARIloadScheduleByMonthchedule: boolean;
  useSameMonthlyLoading: boolean;
  loadScheduleAllMonths: number[];    // 11 values: % loading at each load point (0-100%)
  loadScheduleByMonth: number[][];    // 12×11 matrix: monthly variation
  loadAtPercent: number[];            // load points, default [25, 50, 75, 100]
  kWPerTonAtLoad: number[];           // performance curve values matching loadAtPercent
}
```

**Performance curve**: `loadAtPercent` and `kWPerTonAtLoad` are parallel arrays defining the chiller's part-load efficiency curve. When `isFullLoadEfficiencyKnown` is false, the user enters the full performance curve directly.

**Load schedule**: The Suite API expects percent loading across 11 load bins (0%, 10%, 20%, ... 100%). `useSameMonthlyLoading = true` means the same `loadScheduleAllMonths` profile applies to all 12 months. When `false`, each month in `loadScheduleByMonth[monthIndex]` has its own 11-bin profile.

## WeeklyOperatingSchedule

```typescript
{
  useSameSchedule: boolean;
  days: DayScheduleData[];      // 7 entries, Mon–Sun
  hoursOnMonToSun?: number[];   // computed: hours per day (0–24)
}

// DayScheduleData
{
  off: boolean;
  start: number;    // hour 0–23
  end: number;      // hour 1–24
  allDay: boolean;
}
```

`hoursOnMonToSun` is computed by `getHoursOnMonToSun()` and stored on the object. It is the value actually used by the energy simulation.

## MonthlyOperatingSchedule

## Modification (EEM Modification)

Each modification represents a named scenario. EEM fields are optional objects — only the fields for the chosen EEM(s) will be populated.

```typescript
{
  id: string;
  name: string;
  notes?: string;
  setupDone: boolean;

  // EEM fields (all optional — populated only if that EEM is applied)
  increaseChilledWaterTemp?: IncreaseChilledWaterTempModification;
  decreaseCondenserWaterTemp?: DecreaseCondenserWaterTempModification;
  useSlidingCondenserWaterTemp?: SlidingCondenserWaterTempModification;
  applyVariableSpeedControls?: VariableSpeedControlsModification;
  replaceChillers?: ReplaceChillerModification;
  upgradeCoolingTowerFans?: UpgradeCoolingTowerFansModification;
  useFreeCooling?: UseFreeCoolingModification;
  replaceChillerRefrigerant?: ReplaceChillerRefrigerantModification;
  installVSDOnCentrifugalCompressors?: InstallVSDModification;
}
```

EEM display names are defined in `EEM_LABELS` in [process-cooling-constants.ts](../constants/process-cooling-constants.ts).

## Units of Measure

### Storage Convention

All data is persisted in **Imperial** units. The `existingDataUnits` field tracks what units were in effect when the data was saved (always `'Imperial'` for new assessments).

### Conversion

`ConvertProcessCoolingService.convertProcessCooling(assessment, oldSettings, newSettings)` walks every numeric field and converts in-place using the shared `ConvertValue` utility. This is called in two situations:

1. **On first load** — if the facility-level settings are Metric and the assessment was stored in Imperial
2. **When the user changes units** — triggered from `SystemBasicsComponent`

### Validators

`process-cooling-validation-rules.ts` defines all form field min/max bounds. `SystemInformationFormService` applies these as Angular `Validators.min()` / `Validators.max()` when building each reactive form. The bounds for temperature, flow rate, capacity, etc. are different for Imperial vs. Metric and are selected based on the active `Settings.unitsOfMeasure`.

## Defaults

`getDefaultProcessCoolingAssessment(settings)` in [process-cooling-constants.ts](../constants/process-cooling-constants.ts) returns a fully populated default assessment. Notable defaults:

- Condenser cooling method: Water (`0`)
- Chilled water supply temp: 44°F
- Annual operating hours: 8,760 (continuous)
- One default chiller (centrifugal, 1000 tons, 0.65 kW/ton)
- Weekly schedule: Mon–Fri 8am–5pm
- CO₂ savings ZIP code: `37830` (Oak Ridge, TN — the DOE location)
